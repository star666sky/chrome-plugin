# AI 评审反馈与复审设计

## 目标

让用户可以纠正 AI 的单条评审意见，也可以补充整次评审要求，并让 AI 基于当前 PR 代码重新判断，而不是只生成一段解释。

## 交互设计

### 单条意见反馈

- 每条评审意见右上角增加“反馈”按钮。
- 点击后在该意见内部展开反馈编辑器，不改变详情面板位置。
- 编辑器提供“结论不准确”“遗漏上下文”“建议不合适”三个可选标签、文本输入框、“重新审查此问题”和“取消”。
- 提交后仅锁定当前编辑器，其他面板仍可浏览。
- AI 返回 `confirmed`、`revised` 或 `dismissed`：
  - `confirmed`：保留意见并展示复审说明。
  - `revised`：用修订后的严重级别、标题、说明和建议更新该意见。
  - `dismissed`：保留卡片作为记录，但标记“已撤回”，不再计入紧急/建议数量。
- 每轮反馈与 AI 回复保存在该意见下，允许继续反馈。

### 整体反馈

- 详情面板底部固定一个“补充审查”入口，独立于可滚动意见列表。
- 展开后输入补充要求并点击“重新审查整个 PR”。
- AI 重新读取当前 PR 元数据和 diff，把用户反馈与上一版发现作为补充上下文，重新输出完整 findings。
- 完整复审生成一条新的历史记录，旧记录不覆盖，便于比较。

## 数据与后台链路

- 单条复审消息携带当前历史记录 ID、意见索引、反馈标签和反馈文本。
- 后台从本地历史记录获取原始意见，重新拉取当前 PR diff，只截取目标文件对应的 diff；找不到时使用受长度限制的完整 diff。
- 单条复审结果更新原历史记录，并重新计算紧急/建议数量。
- 整体复审复用现有分片评审链路，为每个分片提示词增加用户反馈和上一版发现摘要，并保存为新历史记录。
- 单条意见新增 `reviewStatus` 和 `feedbackRounds` 字段；旧历史数据默认视为 `active`，保持兼容。

## 错误与边界

- 空反馈不能提交。
- 单条复审期间禁用当前编辑器；整体复审期间沿用主面板评审中状态。
- 历史记录不存在、意见索引失效、当前 URL 与记录 PR 不匹配时明确报错，不更新存储。
- AI JSON 异常或网络失败时保留用户输入，允许再次提交。
- 已撤回意见仍可查看反馈过程，但不参与汇总计数。

## 验证范围

这是自用插件，按用户要求不新增自动化测试。实施后运行现有语法检查、后台模块语法检查和 CSS 括号检查，并人工验证单条确认/修订/撤回、整体复审、失败重试与历史恢复。

## 详情布局优化

- 单条反馈编辑器不再作为意见卡片的末尾内容；它占用详情面板底部的固定工作区，并在打开时临时替代“补充审查”入口。
- 固定工作区显示所选意见的严重级别、标题和文件位置，用户无需滚动即可确认反馈对象。
- 意见列表保持唯一滚动区，当前被反馈的意见只保留高亮状态和“收起”按钮。
- 详情汇总栏压缩为紧凑的四列统计；意见头部改为第一行“严重级别 + 反馈按钮”、第二行独占文件路径，避免长路径挤压严重级别。
