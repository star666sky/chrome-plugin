# 仓库协作约定

## 项目定位

本仓库是个人自用的 Chrome 插件集合，以快速迭代、直接可用和最小维护成本为优先，不按正式产品项目执行完整研发流程。

## 直接实现

- 用户提出明确需求后，先做最少量的现状检查，然后直接修改实现并完成必要验证。
- 对不影响需求方向的细节自行采用合理默认值，不因轻微歧义暂停等待确认。
- 除非用户明确要求，不进行需求评审、方案评审、头脑风暴、设计确认或多轮计划确认。
- 只修改与当前需求直接相关的内容，不顺带重构、扩展功能或引入新的抽象层。

## 不产出设计文档

- 除非用户明确要求，不创建需求说明、设计文档、技术方案、实施计划、变更记录或复盘文档。
- 不创建或恢复 `docs`、`specs`、`plans` 等仅用于研发流程记录的目录。
- 只有当使用方式、安装步骤或配置方式实际变化时，才按需更新现有 README。

## 默认不写测试

- 除非用户明确要求，不新增、恢复或维护单元测试、集成测试、端到端测试及其目录。
- 不为了测试而引入测试框架、测试依赖、Mock、fixture 或辅助脚本。
- 验证优先采用与改动相匹配的最小方式，例如语法检查、构建检查、清单和引用完整性检查、目标场景冒烟验证。

## 安全与版本控制

- 涉及删除用户数据、不可逆操作、外部系统写入或会明显改变需求结果的关键歧义时，仍需先确认。
- 未经用户在当前任务中明确授权，不执行 `git add`、`git commit`、建分支、打标签、创建 PR 或推送远端。
